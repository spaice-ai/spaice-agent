# Code Review: `spaice_agent/memory/continuity.py`

## Factual errors

1. **`read_latest()` section parser breaks on H1 heading** (line 197-199)
   ```python
   elif line.startswith("## "):
       # Any OTHER H2 heading stops processing further.
       # H1 (# LATEST ...) is the document title — ignore it, don't break.
       break
   ```
   **Bug**: The comment claims H1 is ignored, but the code breaks on *any* `## ` heading. The document title `# LATEST — continuity pickup point` starts with `# ` (one hash), not `## `, so it won't trigger this break. However, the logic is still wrong: the code should continue parsing if it encounters an unknown H2, not break. If a future edit adds `## Notes` after `## Next step`, parsing stops prematurely.

2. **Timestamp extraction regex missing** (line 207-211)
   The timestamp line format is `Last updated: {block.ts}`, but the parser does a naive string prefix match. If the file is hand-edited to `Last updated: 2026-05-03T12:00:00+00:00 (manual edit)`, the extracted `ts` includes the trailing text. Should use `.split()[0]` or regex to extract ISO 8601 cleanly.

## Architectural weaknesses

1. **No validation of parsed ContinuityBlock fields**
   `read_latest()` returns a `ContinuityBlock` even if all fields are empty strings. Caller has no way to distinguish "file exists but is corrupted" from "file is valid but empty". Should return `None` if critical fields (goal, next_step) are missing or TODO markers.

2. **`generate_latest()` fusion logic is brittle**
   - If `_top_inbox_summaries()` returns 10 items and `_top_dashboard_todos()` returns 5, `open_threads` has 15+ items (session threads + inbox + todos). No deduplication or length cap. The `next_step` logic picks `open_threads[0]`, which might be a stale session thread instead of the most urgent inbox item.
   - **Ordering assumption**: Session threads come first, then inbox, then todos. But session threads might be days old while inbox has urgent items from 5 minutes ago. Should sort by recency or priority.

3. **Skeleton fallback is too minimal**
   When no session summary exists, `progress = "TODO: Record progress"` and `next_step = "TODO: Identify next action"`. But `open_threads` still gets populated from inbox/dashboards. The skeleton should either be *fully* empty (no threads) or *fully* populated (infer goal from inbox). Current hybrid state is confusing.

## Incomplete specifications

1. **Framework spec: "editable by agent but regenerated on mine"**
   The generated file header says `<!-- GENERATED by spaice-agent continuity; editable by agent but regenerated on mine -->`, but there's no logic to preserve manual edits. If an agent appends `## Notes` section, the next `write_latest()` call obliterates it. Either:
   - Implement merge logic (read existing, preserve unknown sections), or
   - Change comment to `do not edit` (like dashboards.py).

2. **Framework spec: "No LLM call — pure template + file reads"**
   ✅ Correctly implemented. But spec says continuity module "formalises the convention" — where's the validation that LATEST.md follows the convention? Should have a `validate_latest()` function that checks for required sections.

3. **Framework spec: "ts_generated in UTC ISO 8601"**
   ✅ Uses `datetime.now(timezone.utc).isoformat()`. But `read_latest()` doesn't validate the timestamp format. If file is hand-edited to `Last updated: yesterday`, parsing succeeds with `ts="yesterday"`.

## Risk omissions

1. **Race condition in `write_latest()`**
   ```python
   tmp.write_text(content, encoding="utf-8")
   os.replace(tmp, target)
   ```
   If two processes call `write_latest()` concurrently, both write to `.tmp`, then both call `os.replace()`. The second replace wins, but the first process's write is lost. Should use PID-based temp file: `LATEST.{os.getpid()}.tmp`.

2. **No error handling for `os.replace()` failure**
   On Windows, `os.replace()` can fail if `target` is open in another process (e.g., editor). Should catch `OSError` and log/retry.

3. **`_most_recent_session_file()` assumes mtime is reliable**
   If filesystem doesn't preserve mtime (network mount, Docker volume), sorting by mtime is wrong. Should parse filename `YYYY-MM-DD-<session_id>.md` and sort by date prefix.

4. **`_parse_session_summary()` has no error recovery**
   If session file is truncated mid-write (disk full), `read_text()` succeeds but content is incomplete. Parser returns partial data with no indication of corruption. Should validate that all expected sections are present.

## Implementation pitfalls

1. **Regex-free parsing is fragile**
   ```python
   if stripped.startswith("- ") and stripped != "- (none)":
       open_threads.append(stripped[2:].strip())
   ```
   If a thread contains `- (none) of the above`, it's incorrectly filtered out. Should check `stripped == "- (none)"` exactly, not substring match.

2. **`_top_inbox_summaries()` YAML frontmatter stripping is naive**
   ```python
   if text.startswith("---"):
       idx = text.find("---", 3)
       if idx != -1:
           text = text[idx + 3:]
   ```
   If YAML contains `---` in a multiline string, this breaks. Should use proper YAML parser or regex `^---\n.*?\n---\n` with `re.DOTALL`.

3. **`_top_dashboard_todos()` stops at first `top` todos across all files**
   ```python
   if len(todo_lines) >= top:
       break
   ```
   If `projects.md` has 5 todos, the loop never checks `inbox.md`. Should collect `top` todos *per file*, then take the first `top` overall.

4. **No handling of symlinks**
   `_most_recent_session_file()` uses `glob("*.md")`, which follows symlinks. If `_archive/sessions/latest.md` is a symlink to an external file, `stat().st_mtime` might fail or return wrong time.

## Test coverage gaps

1. **No test for concurrent writes**
   Two threads calling `write_latest()` simultaneously could corrupt the file. Need a test with `threading` or `multiprocessing`.

2. **No test for malformed LATEST.md**
   - Missing `## Goal` section
   - Sections in wrong order (`## Next step` before `## Goal`)
   - Duplicate sections (`## Goal` appears twice)
   - Non-UTF-8 encoding

3. **No test for `generate_latest()` with stale session**
   If most recent session is 30 days old, should it still be used? Or should there be a staleness threshold?

4. **No test for empty `open_threads` rendering**
   When `open_threads=[]`, output is `- (none)`. But `read_latest()` filters out `- (none)`, so roundtrip loses the "no threads" signal. Test should verify `len(parsed.open_threads) == 0`.

5. **No test for `_top_inbox_summaries()` with non-UTF-8 files**
   If inbox file is binary or Latin-1 encoded, `read_text(encoding="utf-8")` raises `UnicodeDecodeError`. Test should verify graceful skip.

6. **No test for `_parse_session_summary()` with nested lists**
   ```markdown
   ## Outstanding threads
   - Thread A
     - Sub-item A1
   - Thread B
   ```
   Current parser appends `Sub-item A1` as a top-level thread. Should only capture top-level bullets.

## Verdict

**Ship with fixes** — 2 blockers, 3 majors, 5 minors.

**Blockers** (must fix before merge):
1. **Race condition in `write_latest()`**: Use PID-based temp file to prevent concurrent write corruption.
2. **Section parser break logic**: Fix `read_latest()` to continue parsing unknown H2 sections instead of breaking, preventing data loss if file is extended.

**Majors** (fix in follow-up PR within 1 sprint):
3. **Timestamp extraction**: Validate ISO 8601 format in `read_latest()`, return `None` if malformed.
4. **YAML frontmatter stripping**: Use proper YAML parser or anchored regex to avoid breaking on `---` in content.
5. **Stale session handling**: Add 7-day staleness threshold in `generate_latest()`; ignore sessions older than that.

**Minors** (technical debt, fix opportunistically):
6. Validate `ContinuityBlock` fields in `read_latest()`, return `None` if critical fields missing.
7. Deduplicate/cap `open_threads` length (max 20 items).
8. Add `validate_latest()` function per framework spec.
9. Handle `os.replace()` failure on Windows.
10. Fix `_top_dashboard_todos()` to collect per-file before global limit.

The core design is sound and matches framework spec. Atomic writes are correct (modulo race condition). No LLM calls, pure filesystem reads as required. Main risks are concurrency and parser brittleness, both fixable with targeted patches. Tests cover happy path but miss corruption/concurrency cases — add those in dedicated `test_continuity.py` before Phase 1C ships.
